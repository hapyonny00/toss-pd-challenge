# 제출용 3대 질문 답안 — Toss PD Challenge 2026

과제: 같은 회사 동료 6명이 다음 주까지 1시간 회의를 잡는 경험 설계
결과물: 반응형 웹 프로토타입 (React + 실제 [TDS mobile](https://tossmini-docs.toss.im/tds-mobile/start/) — 375부터 1440까지)

---

## 1. 발견한 사용자 문제

일정 조율 도구들은 "빈 시간 찾기"를 문제로 정의하지만, 캘린더가 연동된 회사에서
빈 시간 계산은 이미 기계가 하고 있어요. 진짜 고통은 그다음에 와요.

> **모두가 되는 시간이 없을 때, '누굴 포기할지'라는 판단과 미안함을
> 주최자가 혼자 떠안는 것.**

관찰한 단서 4가지:
- 조율 메시지에는 "죄송한데", "괜찮으시면"이 반복돼요 — 조율은 감정 노동이에요.
- 참석자 6명의 무게는 같지 않아요. 팀장이 빠진 회의와 선택 참석자가 빠진 회의는 다른 회의예요.
- 응답자는 바빠요. 설문형 도구(전 시간대 체크)는 응답 자체가 새 일이 돼요.
- 조율이 끝나도 일이 남아요 — 캘린더 등록, 회의실 예약, 초대, 알림. **조율의 끝은 확정이 아니라 캘린더예요.**

## 2. 해결 방안

**시스템이 제약의 무게를 흡수해, 주최자가 판단도 죄책감도 지지 않게 해요.**

1. **묻지 말고 아는 걸 써요** — 캘린더 free/busy는 자동으로 읽어요(제목은 안 봐요).
   사람에게는 캘린더가 모르는 것(꼭/선택, 피하고 싶은 시간)만 물어요.
2. **응답은 네거티브 선택 10초** — "안 되는 시간만" 탭해요. 기본값이 '다 괜찮아요'예요.
3. **후보 나열이 아니라 단일 추천** — 비교를 시키는 순간 '누굴 포기할지'가 주최자 몫이 돼요.
   추천 근거(누가 오는지)를 아바타로 보여주고, 판단은 시스템이 져요.
4. **전원 참석은 '거래'로** — "박지후님 회의를 30분만 옮기면 6명 전원이 모여요."
   미안한 부탁 메시지는 시스템이 대필하고, 받는 쪽은 버튼 두 개만 눌러요.
5. **확정 = 등록** — 확정 버튼을 누르는 순간 6명 캘린더 등록·회의실 예약·화상 링크·알림까지
   시스템이 끝내요. 화면은 등록 '과정'이 아니라 등록된 '결과'(이벤트 카드)를 보여줘요.
   개인 캘린더(Google·Outlook·Apple)로 내보내기는 실제로 동작해요.

## 3. 그렇게 설계한 이유

- **웹 퍼스트**: 회사 일정의 스타팅 포인트는 웹(업무 환경)이에요. 데스크톱에서는
  본문 옆에 맥락이 붙는 2컬럼, 모바일에서는 풀블리드로 같은 플로우가 이어져요.
- **One Thing per One Page**: 화면마다 주 결정 하나, 파란 버튼 하나. 조율처럼 불안한
  과업일수록 화면이 조용해야 해요.
- **막다른 길 금지**: 추천이 다 별로면 거래로, 동료가 다 안 되면 되는 시간 입력으로,
  대기가 길어지면 5명 확정으로 — 모든 화면에 출구가 있어요.
- **캘린더 등록을 '기능'이 아니라 '소멸'로 설계**: 가장 편한 등록은 등록 화면이 없는 것이에요.
  대신 결과를 검증 가능하게 — 이벤트 카드는 캘린더에 들어간 모습 그대로이고, 제목만
  자동 생성이라 누르면 바로 고칠 수 있어요. 주간 뷰에 이벤트가 내려앉는 모먼트가
  '조율이 끝났다'를 공간으로 확인시켜요.
- **변경도 처음보다 쉽게**: 캘린더 상태를 이미 아니까 대안을 즉시 제시하고, 각 대안의
  손해("지금보다 1명 줄어요")를 정직하게 붙여요. 바꾸면 캘린더·회의실·알림을 시스템이 수습해요.
- **라이팅이 곧 UX**: 토스 라이팅 원칙(해요체·능동·긍정·캐주얼 경어) 위에, "누구의 잘못도
  아니게"라는 제품 원칙을 더했어요. docs/UX_WRITING.md 참고.
- **실제 디자인 시스템**: 색·버튼·리스트·배지 전부 토스가 공개한 TDS mobile 컴포넌트와
  시맨틱 컬러 토큰을 그대로 가져다 썼어요. 흉내 낸 스타일이 아니라 토스 프로덕트가 실제로
  쓰는 조형 문법이라, 시각 완성도 판단 기준(컴포넌트 사용 적절성)에 그대로 부합해요.
