## 0. 이 문서를 읽는 법

이 서비스는 **"같은 회사 동료 6명이 다음 주 안에 1시간 회의를 잡는다"**는 단일 과제를 푼다.
핵심 문제 정의는 리포 전체와 동일하다 — 바꾸지 않는다.

> 일정 조율의 진짜 고통은 빈 시간 찾기가 아니라, 모두가 안 될 때 **'누굴 포기할지'를 주최자가 혼자·미안하게 떠안는 것**이다.
> 시스템이 제약의 무게를 흡수해, 주최자가 판단도 죄책감도 지지 않게 한다.

이 명세서가 기존 웹 프로토타입([`gcal-prototype`](../gcal-prototype/))과 다른 점은 딱 세 가지다.

1. **표현 매체가 삼성 One UI** — 데스크톱 캘린더가 아니라 "이동 중에도 회의를 잡는" 모바일 네이티브 앱이 1순위. 하단 집중 구조·거대 타이틀·바텀시트가 뼈대.
2. **라이트/다크 동시 설계** — 모든 토큰과 컴포넌트가 2모드로 정의된다. 다크는 AMOLED true-black 기준.
3. **반응형 3-tier** — 모바일(360dp) → 폴더블/태블릿(600dp) → 데스크톱 웹(1024px+). 같은 정보 구조가 폼팩터별로 재배치된다.

## 1. 디자인 원칙

### 1.1 기존 설계 6원칙 (불변)
`CLAUDE.md`에서 그대로 승계한다. One UI로 옮겨도 이 6개는 결정의 기준점이다.

1. **막다른 길 금지** — 모든 화면에 출구가 있다.
2. **묻지 말고 아는 걸 쓴다** — 캘린더 free/busy는 자동, 캘린더가 모르는 선호만 최소한으로.
3. **One Thing per One Page** — 한 화면(=한 시트)에 주 결정 하나. 주 버튼 하나.
4. **강조 리듬 3단** — 히어로 > 거래 > 차선. 강조 요소는 화면당 하나.
5. **라이팅은 토스 원칙대로** — 해요체·능동·긍정·다정한 구어체. 금지어(가중치·최적화·교집합) 금지.
6. **확정 = 등록** — 캘린더 등록은 과정이 아니라 자동으로 끝난 결과.

> ⚠️ 원칙 5 때문에 UI 노출 문구에는 "가중치/필수/선택" 대신 **"꼭 참석 / 가능하면"**을 쓴다.
> 이 명세서 본문(설명)에서는 이해를 위해 "가중치"를 쓰되, **화면에 찍히는 카피는 전부 순화형**이다.

### 1.2 One UI 적응 원칙 (신규 4개)
삼성 One UI 8.5의 철학을 이 서비스에 이식하며 세운 원칙.

**P1. 하단 집중 (Bottom-heavy reach)**
One UI의 근본 사상은 화면을 **위=보는 영역(Viewing area) / 아래=만지는 영역(Interaction area)**으로 나누는 것이다.
- 화면 상단 40%: 거대 타이틀·컨텍스트·주간 캘린더(읽기 전용, 손가락 안 닿아도 됨).
- 화면 하단 60%: 리스트·컨트롤·주 버튼·바텀시트(엄지로 다 조작 가능).
- 주 액션(FAB, 주 버튼, 시트 확정 버튼)은 **항상 화면 최하단**. 한 손 조작 시 엄지 스윕 존(safe reach) 안.

**P2. 시트 우선 (Sheet over Page)**
플로우 단계 전환은 풀페이지 이동이 아니라 **바텀시트의 높이 변화**로 표현한다.
- 캘린더(맥락)는 항상 위에 남는다. 시트가 peek(추천 1개) → half(추천 3개) → full(전체 이유)로 자라며 정보가 드러난다.
- 이유: 주최자가 "내 결정이 캘린더에 어떻게 반영되는지"를 한 화면에서 계속 보게 하려고. 맥락 상실 방지.

**P3. 거대 타이틀 → 앱바 붕괴 (Collapsing large title)**
각 시트/화면 최상단에 24–28px 거대 타이틀. 콘텐츠를 스크롤하면 타이틀이 축소되며 상단 앱바(20px)로 붙는다. One UI의 시그니처 제스처 감각을 그대로.

**P4. 곡률로 위계 (Radius as hierarchy)**
One UI는 그림자를 거의 안 쓴다. 대신 **곡률(26px 카드)과 표면 명도 차이**로 요소를 띄운다.
- 라이트: 회색 배경 위 흰 카드.
- 다크: 검정 배경 위 짙은 회색 카드(#1B1B1D). 그림자 대신 표면 명도로 elevation을 표현.

### 1.3 두 시스템의 역할 분담
이 서비스는 **삼성 One UI의 그릇**에 **토스의 화법**을 담는 하이브리드다. 충돌하지 않게 역할을 못박는다.

| 레이어 | 담당 시스템 | 구체 |
|---|---|---|
| 폼팩터·구조·컴포넌트 형태 | **One UI 8.5** | 하단 집중, 26px 곡률, 바텀시트, FAB, 스위치, 칩, 거대 타이틀 |
| 색·정보 위계 판단 | **One UI + TDS 관점** | 아래 §2 컬러. 히트맵 계층은 TDS의 "면·선·점" 위계 사고를 차용 |
| 문장·톤·마이크로카피 | **토스 라이팅(TWS)** | 해요체, 다정한 구어체, §7 전문 |

## 2. 디자인 토큰

모든 토큰은 **역할(semantic) 이름**으로 정의한다. 컴포넌트는 raw hex를 직접 참조하지 않는다.
구현 시 CSS custom property(`--color-...`) 또는 디자인 툴 Variable로 1:1 매핑. 라이트/다크는 같은 토큰 이름에 값만 스왑.

> **다크모드 3대 규칙**
> 1. 순수 검정(`#000`)은 **base 배경에만**. 카드는 반드시 `#1B1B1D` 이상으로 띄운다(명도 elevation).

### 2.4 Radius · Spacing · Elevation · Motion

**Corner radius** (One UI는 크고 둥글다)
| 토큰 | 값 | 용도 |
|---|---|---|
| `radius/sheet` | 28px | 바텀시트 상단 좌우 |
| `radius/card` | 26px | 카드·컨테이너 |
| `radius/field` | 26px | 검색·입력 캡슐 |
| `radius/button` | 18px | 사각 버튼 |
| `radius/cell` | 12px | 히트맵 셀·리스트 눌림 하이라이트 |
| `radius/chip` | 999px | 칩·세그먼트·아바타·FAB(원형) |

**Spacing** (4px base) — `4 · 8 · 12 · 16 · 20 · 24 · 28 · 32`
- 모바일 화면 좌우 여백(safe gutter): **24px**
- 카드 내부 패딩: **20px**(상하) / **24px**(좌우)
- 리스트 행 높이: 최소 **56px**, 아바타 포함 시 **64px**
- 터치 타깃 최소: **48×48dp**(원칙)

**Elevation** (라이트만 그림자, 다크는 표면명도)
| 레벨 | 라이트 그림자 | 다크 |
|---|---|---|
| `elev/card` | 없음(표면색으로 분리) | 없음 |
| `elev/sheet` | `0 -4px 24px rgba(0,0,0,.10)` | 없음 + 상단 1px `border/hairline` |
| `elev/fab` | `0 4px 16px rgba(56,122,255,.32)` | `0 4px 16px rgba(0,0,0,.5)` |
| `elev/dialog` | `0 8px 40px rgba(0,0,0,.18)` | `0 8px 40px rgba(0,0,0,.6)` |

**Motion** — One UI 특유의 스프링감
| 토큰 | duration | easing | 용도 |
|---|---|---|---|
| `motion/quick` | 200ms | `cubic-bezier(.4,0,.2,1)` | 칩·스위치 토글, 눌림 |
| `motion/standard` | 300ms | `cubic-bezier(.22,1,.36,1)` | 시트 높이 변화, 페이지 전환 |
| `motion/emphasized` | 420ms | `cubic-bezier(.34,1.56,.64,1)` | 확정 시 이벤트 카드 착지(살짝 오버슛) |
| `motion/heatmap-reveal` | 500ms | `cubic-bezier(.22,1,.36,1)` | 히트맵 레이어가 켜지며 셀이 순차 페이드(stagger 12ms/셀) |
| `reduced-motion` | 0ms 대체 | opacity만 | 시스템 접근성 설정 시 이동 애니메이션 제거, 페이드만 |

**바텀시트 3-스냅** (P2의 핵심)
- `peek` (높이 40%): 추천 1위 카드 1장만 노출. 캘린더 최대 노출.
- `half` (높이 62%): 추천 3장 + "더보기". 기본 상태.
- `full` (높이 92%): 각 추천의 상세 이유·히트맵 범례·엣지케이스 진입.
- 드래그 핸들로 스냅 이동. 스와이프 다운으로 한 단계씩 축소. 스크림 탭으로 닫힘.

### 3.3 데스크톱 재배치 규칙
- 바텀시트 → **우측 고정 패널**(폭 344px, `surface/1`, 좌측 1px `border/hairline`)로 승격.
- FAB → 사이드바 상단 **"회의 잡기" 주 버튼**으로 승격.
- 거대 타이틀 붕괴 제스처 → 데스크톱은 패널 헤더 고정(붕괴 없음).
- 히트맵 셀 hover 시 툴팁 노출(모바일은 long-press).
- 자세한 데스크톱 스펙은 `gcal-prototype`을 계승하되 컬러만 §2로 교체.

## 4. 컴포넌트 라이브러리

이 서비스가 실제로 쓰는 One UI 컴포넌트 12종. 각 컴포넌트는 라이트/다크·상태·치수를 명시한다.

### C1. Large Title Header (거대 타이틀 헤더)
- 구성: `large-title`(28/700) + 우측 액션 아이콘(24dp). 스크롤 오프셋 0–56px 구간에서 타이틀이 28→20px로 선형 축소, 위치가 앱바로 이동.
- 라이트: 배경 `surface/base`, 글자 `text/primary`. 다크: 배경 `#000`, 글자 `#FFF`.
- 붕괴 완료 시 앱바 하단에 `border/hairline` 1px 등장(스크롤 그림자 대체).

### C2. Bottom Sheet (바텀시트)
- 상단 좌우 `radius/sheet`(28px), 상단 중앙 드래그 핸들(32×4px, `border/strong`, radius 999).
- 배경 `surface/sheet`, 라이트 그림자 `elev/sheet` / 다크 상단 hairline.
- 3-스냅(§3.2). 최하단에 주 액션 바가 시트에 고정(스크롤과 무관하게 항상 보임).

### C3. Filled Button (주 버튼)
- 높이 56px, `radius/button`(18px), 라벨 `type/button`(16/600) 중앙.
- 라이트: 배경 `accent/fill`, 글자 `accent/on-fill`. 눌림 `accent/fill-press` + scale(.98).
- 다크: 배경 `accent/fill`(#3B76F5), 글자 흰색.
- 비활성 회피(원칙) — 항상 활성, 조건 미충족 시 눌렀을 때 인라인 안내.

### C4. Tonal / Ghost Button (보조 버튼)
- Tonal: 배경 `accent/tint`, 글자 `accent/text`. Ghost: 투명 배경, 글자 `text/secondary`, 눌림 시 `surface/3`.
- 높이 52px. "다른 시간 볼게요" 등 보조 동작.

### C5. Switch (스위치) — 클릭① 가중치용
- One UI 스위치: 트랙 52×32px, 노브 28px, radius 999.
- OFF: 트랙 `surface/3`, 노브 `#FFF`(라이트)/`#B0B0B6`(다크). ON: 트랙 `accent/fill`, 노브 흰색.
- 토글 `motion/quick`. 노브 이동 + 트랙 색 크로스페이드.

### C6. Chip / Segmented Toggle (칩·세그먼트) — 클릭① 대안
- 세그먼트: 컨테이너 `surface/2`, radius 999, 내부 2px 패딩. 선택 세그먼트 흰 배경(`surface/1`)+`accent/text` 글자+`elev/card`. 미선택 `text/secondary`.
- "꼭 참석 / 가능하면" 2분기 세그먼트로 사용. 높이 40px.

### C7. Attendee Row (참석자 행)
- 높이 64px. 좌측 아바타(40dp 원형, 이니셜, 개인 색), 중앙 이름(`list-title`)+역할(`subtitle`), 우측 세그먼트(C6).
- 행간 구분: `border/hairline` 1px(마지막 행 제외). 눌림 시 행 전체 `surface/3` 하이라이트(radius/cell).

### C8. Heatmap Cell (히트맵 셀) — 클릭②의 최소 단위
- 주간 그리드의 1시간 슬롯. 폭=열폭, 높이 44–56px, `radius/cell`(12px), 셀 간 gap 3px.
- 상태별 채색은 §2 `state/*`. 위계는 §5.2 상세.
- 추천 셀에만 중앙 `추천 N` 라벨(`caption`, `accent/text`). 부담 셀 우상단 `dot/burden`(6px).

### C9. Recommendation Card (추천 슬롯 카드) — 클릭③
- `radius/card`(26px), 패딩 20/24, 배경 `surface/1`, 보더 `border/hairline`.
- 구성: 상단 랭크 칩(`추천 1`, `accent/tint`+`accent/text`) → 시간(`title` 22/700) → 이유 2줄(`subtitle`).
- 선택(포커스) 시 보더 → `border/accent` 1.5px + 배경 `accent/tint`.

### C10. FAB (플로팅 액션 버튼)
- 56dp 원형, 배경 `accent/fill`, 아이콘 `+`(24dp, 흰색), `elev/fab`.
- 위치: 화면 우하단, 우측 24px·하단 24px(제스처바 위). 스크롤 시 축소(라벨 숨김) 옵션.

### C11. Toast (토스트)
- One UI 토스트: 하단 캡슐, 배경 다크 반투명(`rgba(28,28,30,.92)`), 글자 흰색, `type/subtitle`, radius 999, 좌측 상태 아이콘.
- 하단에서 16px 위, 2.6s 후 페이드. 라이트/다크 공통(어두운 캡슐).

### C12. Cold-start Suggestion Card (자동 제안 카드) — §6
- 진입 시 하단에서 rise하는 카드. `radius/card`, 배경 `surface/1`, `elev/sheet`.
- 상단 태그 칩("캘린더를 읽어봤어요", `accent/tint`) → 제목(`list-title`) → 설명(`subtitle`) → 2버튼(Tonal 확인 / Ghost 거절).

### 5.2 클릭② — 선호 히트맵 레이어 (하이라이트 구간)

이 화면이 이 서비스의 **비주얼 클라이맥스**다. 6명 일정이 겹쳐도 복잡해 보이지 않게 하는 것이 전부다.

**[IA]** "되는 시간 찾기" 탭 → ① 캘린더 위에 히트맵 레이어가 `motion/heatmap-reveal`로 순차 점등 → ② 시트가 `peek`로 내려가 캘린더를 최대 노출 → ③ 시트 핸들 위에 "이 시간이 좋아요" + 추천 1위 카드.

**[정보 계층 — 3겹 규칙 (TDS "면·선·점" 관점)]**
6명의 개별 색을 **버린다.** 개인 색으로 6줄을 쌓으면 그게 색종이 더미다. 대신 **집계된 상태 하나**로 각 셀을 칠하고, 위계를 3개 채널로 분리한다.

1. **면(Fill) = 적합도 4단계** — 셀 배경색 하나로 "이 시간이 얼마나 좋은가"만 말한다.
   - `state/comfy`(연녹) 모두 편해요 > `state/ok`(중립 표면) 가능해요 > `state/hard`(가라앉은 회색) 어려워요.
   - 어려워요 셀은 **채도를 빼서 뒤로 밀어낸다**(눈이 지나치게 함). 좋은 시간이 튀어나오게.
2. **선(Border/Stroke) = 추천** — 상위 3개 슬롯에만 `border/accent` 1.5px + `accent/tint` 배경 + `추천 N` 라벨. 화면 전체에서 **파란 테두리는 정확히 3곳**뿐. 시선이 3초 안에 도착.
3. **점(Dot) = 부담 경고** — 누군가 기피(외근 직후·점심 직후·연속회의)인 셀 우상단에 `dot/burden`(6px 앰버). 꼭 참석자 부담은 7px, 가능하면 참석자 부담은 5px 흐린 회색 점(위계 차등).

> 결과: 셀 하나가 최대 3개 채널(면 1 + 선 0/1 + 점 0/1)만 인코딩. 개별 6명 정보는 **long-press 툴팁**으로 캡슐화 → 평소엔 안 보임, 궁금할 때만.

**[모바일 픽셀]**
- 주간 그리드 셀: 폭 (360−48−타임열40)/5 ≈ 54px, 높이 48px, gap 3px, `radius/cell`.
- `peek` 시트(높이 40%) 상단: 핸들 → "이 시간이 좋아요"(large-title 축소형 20/700) → 추천 1위 카드(C9) 1장 → "추천 더 보기 ⌄".
- 시트를 위로 드래그(half) → 추천 2·3위 카드 등장. full → 각 카드 펼침 + 히트맵 범례 바.
**[상태]**
- 레이어 토글 칩(앱바 우측): "팀의 시간 레이어 · 켜짐". 끄면 내 일정만 보이는 순수 캘린더로(비교용).
- 셀 long-press → 툴팁 "준호 님이 외근 직후예요"(개인명은 여기서만, 그것도 집계 표현).
- 추천 셀 탭 = 추천 카드 탭과 동일 → 확정.
**[라이트/다크]**
- comfy: 라이트 `#E3F5EA` / 다크 `#12301F`(녹색 아주 낮은 명도, 눈부심 방지).
- hard: 라이트 `#ECECEE` / 다크 `#161618`(base보다 살짝 위, "죽은 셀"로 인식되게).
- reco border: 라이트 `#387AFF` / 다크 `#82AAFF`. 부담 도트는 양 모드 `#FFB43A` 고정.
- 범례 바: 라이트 반투명 흰 / 다크 반투명 `#1B1B1D`.
**[카피]**
- 레이어 칩 `팀의 시간 레이어 · 켜짐` / `· 꺼짐`
- 시트 타이틀 `이 시간이 좋아요`
- 시트 안내 `여섯 분의 캘린더와 컨디션을 겹쳐 봤어요. 파란 칸을 바로 눌러도 돼요`
- 범례 `어려워요 · 가능해요 · 모두 편해요 · 추천 · 조금 힘든 시간`
- 툴팁 예 `준호 님이 외근 직후예요` / `점심 직후라 집중이 어려운 시간이에요`
- 더보기 `추천 더 보기`
**[반응형]** 데스크톱: 히트맵이 중앙 큰 캘린더, 추천 카드는 우측 패널에 3장 세로. long-press→hover 툴팁.

### 5.3 클릭③ — 추천 슬롯 카드 (신뢰의 카피)

**[IA]** 시트 `half`에 추천 3장. 각 카드 = 랭크 + 시간 + **추천 이유(왜 이 시간인가)**. 이유가 신뢰를 만든다.
**[모바일 픽셀]** 카드(C9) 3장 세로, 카드 간 gap 10px. 카드 탭 → 확정(5.4).
**[추천 이유 — 알고리즘 투명성을 다정하게 (TWS)]**
이유는 "필수 3명 교집합 + 기피 0" 같은 계산을 **사람 말로 번역**한다. 절대 수식·점수·퍼센트를 노출하지 않는다.

| 랭크 | 시간 | 추천 이유 카피 (TWS) |
|---|---|---|
| 추천 1 | 화요일 오전 10시 | `꼭 참석할 세 분 모두 편한 시간이에요. 여섯 분 전원이 올 수 있어요.` |
| 추천 2 | 수요일 오전 11시 | `여섯 분 전원이 올 수 있어요. 부담스러운 시간이 있는 분도 없어요.` |
| 추천 3 | 화요일 오후 3시 | `전원이 올 수 있지만, 도윤 님은 바로 앞에 회의가 있어 조금 빠듯할 수 있어요.` |

> **카피 원칙 (이유 문장 공식)**
> 1. **좋은 점 먼저, 아쉬운 점 뒤** — "전원 가능해요"(안심) → "다만 도윤 님이 빠듯"(정직).
> 2. **주어는 사람** — "교집합이 3" ✗ → "세 분 모두 편해요" ○.
> 3. **아쉬운 점도 숨기지 않는다** — 3위에 굳이 단점을 적는 건, 시스템이 영업사원이 아니라 참모임을 증명.
> 4. **숫자는 셀 수 있는 것만** — "여섯 분", "세 분"은 OK. "87% 적합"은 금지.

**[상태]** 카드 포커스 시 border/accent. "만약 모두 되는 시간이 없다면?" ghost 링크(엣지 진입, 데모/시연용 상시 노출).
**[라이트/다크]** 카드 `surface/1` + hairline. 랭크칩 `accent/tint`+`accent/text`. 다크 랭크칩 `#16233D`+`#82AAFF`.
**[반응형]** 데스크톱 우측 패널 3장 세로 동일.

### 5.4 확정 — 완료 (확정 = 등록)

**[IA]** 추천/카드 탭 → ① 캘린더의 해당 셀에 이벤트 카드가 `motion/emphasized`로 착지(오버슛) → ② 시트가 완료 상태로 전환(체크 + 요약) → ③ 캘린더 등록 자동 완료.
**[모바일 픽셀]** 시트 중앙: 56dp accent 원형 체크 → 제목(title 22/700) → 요약(subtitle) → (조건부) 추가 안내 박스 → 주 버튼 "완료" + ghost "다른 시간 볼게요".
**[상태]** 확정 즉시 6명에게 초대 발송 + 주최자 캘린더 등록(별도 저장 버튼 없음). "다른 시간 볼게요"로 되돌리기(undo) → 5.3.
**[카피]**
- 제목 `화요일 오전 10시로 잡았어요`
- 요약 `여섯 분 모두에게 초대를 보냈어요. 지민 님 캘린더에도 등록해 뒀어요.`
- 버튼 `완료` / `다른 시간 볼게요`
**[라이트/다크]** 체크 원형 `accent/fill`. 다크도 동일(흰 체크).

### 5.5 엣지케이스 — 타협 스펙 (막다른 길 금지의 정점)

**전제:** 다음 주 안에 6명 전원 1시간이 **물리적으로 0개**. 여기서 "시간이 없습니다"는 **금지어**다. 시스템은 제약을 하나씩 풀며 차선책을 계산해 **3장의 타협 카드**로 내놓는다.

**[완화 축 우선순위 — 회의 목적을 덜 훼손하는 순서]** (상세 로직 §5.5 하단 + 렌더는 prototype)
| 순위 | 완화 축 | 논리 |
|---|---|---|
| 1 | 가능하면 참석자 1명 제외 | 비용 최저. 애초에 '가능하면'으로 분류한 이유가 이 순간을 위해서. |
| 2 | 기피 시간 허용 | 만남은 성사. 기피 강도 약하고 기피 인원 적은 슬롯부터. |
| 3 | 60→30분 단축 | 짧아도 전원이 낫다(킥오프류). |
| 4 | 다음다음 주로 확장 | 긴급하지 않으면 가장 건강한 답. |

**[타협 카드 3장 구조]** 각 카드 = **얻는 것 · 잃는 것 · 원탭 실행**. 잃는 것을 숨기지 않는다.
**[모바일 픽셀]** 시트 full. "다음 주엔 모두 모이기 어려워요 / 대신 이런 방법이 있어요" 헤더 → 카드 3장(각 제목/설명/실행 Tonal 버튼) → 하단 주 버튼 "그다음 주 더 보기".
**[카피]**
- 헤더 `다음 주엔 모두 모이기 어려워요. 대신 이런 방법이 있어요.`
- 서브 `무엇을 얻고 무엇을 내려놓는지 함께 보여드릴게요. 결정은 지민 님이 하면 돼요.`
- 카드1 제목 `민지 님 없이 진행하면` / 설명 `수요일 오전 11시에 모일 수 있어요. 회의 내용은 민지 님에게 자동으로 정리해서 보내드릴게요.` / 버튼 `이 시간으로 잡기`
- 카드2 제목 `화요일 오후 1시는 어때요?` / 설명 `준호 님이 외근 직후라 조금 힘들 수 있는 시간이에요. 양해 메시지를 대신 보내드릴까요?` / 버튼 `메시지 보고 잡기`
- 카드3 제목 `30분으로 줄이면` / 설명 `목요일 오후 2시에 여섯 분 모두 모일 수 있어요.` / 버튼 `30분으로 잡기`
- 최하단 `그다음 주 더 보기`

**[시니어 디테일 2가지]**
1. **미안함의 아웃소싱** — 카드2. 기피 시간에 잡을 때 진짜 비용은 시간이 아니라 감정(준호에게 미안함)이다. 시스템이 **양해 메시지 초안을 대필**한다. 문제 정의("눈치·감정 소모")를 정면으로 타격.
   - 대필 문안(존댓말 예외 허용): `준호님, 화요일 오후 1시가 외근 직후라 힘드실 것 같아 먼저 양해를 구합니다. 꼭 필요한 이야기만 나누고 1시간 안에 마칠게요. — 지민 드림`
2. **제외된 사람의 사후 경험** — 카드1로 민지를 빼는 순간, 민지 입장 스토리가 생긴다. "당신 빼고 잡힘"이 아니라 **"일정이 안 맞아 이번엔 요약으로 대신, 회의록 도착 시 알림"**. 주최자 화면만 닫으면 반쪽 — 타협 대상의 화면까지 설계해야 완성.
   - 민지에게 갈 알림 카피: `이번 회의는 일정이 맞지 않아 참석하지 못했어요. 끝나면 핵심 내용만 정리해서 보내드릴게요.`

**[라이트/다크]** 카드 `surface/1`. 카드2 대필 미리보기 박스는 `surface/2` + 좌측 `accent/text` 세로 강조선.

## 6. 자동 태깅 엔진 (콜드 스타트 해결)

To-Be가 작동하려면 팀원의 '선호/기피'가 있어야 하는데, **바쁜 현업은 입력 안 한다.** 그래서 **입력을 없앤다.** 시스템이 과거 캘린더에서 추론하고, 사용자는 확인만 한다.

### 6.1 추론 신호 (Signal → Inference)
| 신호(캘린더에서 읽는 것) | 추론 | 태그 |
|---|---|---|
| 특정 요일 반복되는 외근/출장(제목 키워드 + 장소 + 종일) | 그 직후 1–2시간 피곤 | 기피(외근 직후) |
| 연속 3개 이상 회의 블록 | 그 뒤 시간 집중 저하 | 기피(연속 회의 직후) |
| 12–13시 반복 비움 | 점심 | 기피(점심 직후 1시간) |
| 장소가 바뀌는 연속 일정 | 이동 시간 필요 | 기피(이동 직후) |
| 특정 시간대 회의가 거의 없음(빈 패턴) | 선호 근무 집중 시간 | 선호 |
| 사용자가 잡아둔 "집중"/"작업" 블록 | 방해 금지 | 기피(집중 시간) |
| 아침 첫 일정 시각 패턴 | 이른 오전 회의 선호/기피 | 선호 또는 기피 |

### 6.2 프라이버시 설계 (핵심)
- 개인의 기피 사유 원문("점심 직후 집중 안 됨")은 **주최자에게 절대 원문 노출 금지.**
- 주최자는 오직 **집계된 히트맵**(면·점)만 본다. long-press 툴팁도 "준호 님이 외근 직후예요" 수준의 **행동 근거**까지만, 감정·사생활 문구는 없음.
- 이유: 부담 없이 솔직한 선호를 등록(=자동 태깅 수용)하게 하려면, 그게 남에게 '평가'로 노출되지 않아야 한다.
- 사이드바/설정 고지 문구: `팀원의 사정은 그대로 보이지 않아요. '조금 어려운 시간'으로만 표시돼요.`

### 6.3 확인 루프 (Confirm, not Configure)
- 시스템이 추론 → **한 가지씩 질문으로 확인**(콜드스타트 카드, 5.0b). 절대 폼을 던지지 않는다.
- 사용자가 "네" → 태그 확정. "아니오" → 그 신호 억제(재학습).
- 회의가 끝난 뒤 사용자가 시간을 자주 바꾸면 → 해당 태그 신뢰도 하향(암묵 학습).

### 6.4 왜 자동 확정하지 않고 확인을 받나 (심사 대비)
- 최종 결정·맥락은 항상 사람에게. 시스템이 모르는 맥락(그날 임원 보고 등)이 늘 존재.
- "먼저 일하고, 확인만 받는다"가 신뢰의 조건. 추천 3개만 주고 자동 확정 안 하는 것과 같은 철학.

## 8. 접근성 & 개발 핸드오프

### 8.1 접근성 체크리스트
- **터치 타깃** 모든 인터랙션 48×48dp 이상. 히트맵 셀은 시각 44px여도 터치 히트박스 48px 확장.
- **대비** 본문 텍스트 AA 4.5:1, 큰 텍스트/아이콘 3:1. §2 액센트 2분리가 이걸 보장(다크에서 파랑 텍스트는 `#82AAFF`).
- **색에만 의존 금지** 히트맵은 색(면)만이 아니라 라벨(추천 N)·점(부담)·범례로 다중 인코딩. 색맹 사용자도 구분 가능.
- **스크린리더** 히트맵 셀 aria-label 예: "화요일 오전 10시, 모두 편한 시간, 추천 1". 세그먼트 role=radiogroup.
- **동적 타입** 시스템 폰트 확대 시 레이아웃 붕괴 없이 리플로우(고정 높이 대신 min-height).
- **모션 감소** `prefers-reduced-motion` 시 §2.4 `reduced-motion` 규칙 — 이동 애니 제거, 페이드만.
- **한 손 사용** P1 하단 집중이 곧 접근성. 주 액션이 엄지 안전 존에.

### 8.2 개발 핸드오프 노트
- **토큰 우선** — 컴포넌트에 raw hex 금지. §2 시맨틱 토큰을 CSS var/디자인 Variable로 등록 후 참조.
- **다크 = 별도 값, 같은 이름** — `[data-theme="dark"]` 스코프에서 토큰 값만 재정의. 컴포넌트 코드는 분기 없음.
- **바텀시트 스냅** — peek/half/full 3-스냅은 스크롤 위치가 아니라 명시적 상태값. 드래그 종료 시 가장 가까운 스냅으로 관성 스냅.
- **히트맵 데이터 구조** — 셀 = `{day, hour, fill: comfy|ok|hard, reco: 1|2|3|null, burden: req|opt|null, tip?}`. 개인 6명 원본은 서버에서 집계 후 이 형태로만 클라이언트 전달(프라이버시 §6.2).
- **날짜 로직 분리** — 다음 주 날짜·초대·ICS 등 계산은 `lib/calendar` 계층에만. 컴포넌트에 날짜 계산 금지(기존 리포 규칙 계승).
- **렌더 확인** — [`prototype.html`](prototype.html)에 라이트/다크·모바일/데스크톱 토글 내장. 헤드리스 크롬 스크린샷으로 4조합 검증 완료(§ 프로토타입 README).
